Redline iOS — Elegant UI/UX Plan (Truth Surface v1)

Dated: 2026-03-02 • Implements SSOT semantics in docs/redline/ios/learning_loop_truth_forcing_plan_v1.md (this doc is execution-level UI/UX detail, not a competing plan).

Attribution is Job #1. Redline is truth-forcing: no silent “fine,” no anchoring defaults, and every user action yields a reversible receipt.

1) UX invariants (must hold everywhere)

  • No hidden states. If an action is unavailable, keep it visible but locked; tap explains why.
  • No anchoring defaults. If the model is weak, do not preselect a project and do not allow swipe-confirm.
  • Receipts over reasoning. Show deterministic evidence tokens and reason chips; never show “because…” text.
  • Everything reversible. Every write yields a receipt + Undo (fast, bottom-of-screen).
  • Speed is part of truth. If pick-time p90 is slow, users will lie (mindless swipes); fix UX first.

Shared primitives (canonical)

3-state chip

Pending Ready Blocked

Forcing-state pill

PICK_REQUIRED FAST_CONFIRM

Reason chips

Show max 2 + “+N”. Example: weak_anchor, quote_unverified.

Suggested chip

Hint-only outline. Tap can prefill selection, but never commits; Apply is required.

2) Forcing states UX contract (truth forcing)

Every item maps to exactly one forcing state and the UI must enforce its constraints. This is the backbone of a learning loop that does not poison itself.

Forcing state Primary user intent UI must enforce Success proof
PICK_REQUIRED Choose the correct project (anti-anchoring). No preselect. “Confirm” locked. Primary CTA is “Pick project”. Suggested is hint-only. Apply commits. Pick-time p90 < 20s; undo_rate < 5%; SIM_PROOF shows locked confirm + explicit picker.
FAST_CONFIRM Confirm a high-confidence assignment quickly. Allow 1-gesture confirm only if evidence tokens exist and substrate is healthy. Still show receipt + Undo. 1-hand confirm; receipt toast present; queue visibly shrinks (real queue_id).
NEEDS_SPLIT Avoid lying (multi-project span). Hard-block assignment. Primary CTA is “Mark multi-project” (v1) and optionally “Replay pipeline.” Card shows Blocked with explicit reason; no accidental assign path exists.
PIPELINE_DEFECT Repair broken substrate, don’t ask user to compensate. Replace card with defect block. Disable assignment. Offer 1-tap repair (idempotent) with status feedback. Repair request_id displayed; success/fail visible; no “done” empty-state while defect exists.

Read-only / writes-locked mode (internal safety posture)

  • If write returns 401/403 invalid_auth, flip once into locked state; disable write controls.
  • Persist a banner: “Writes locked (needs privileged auth). Reads still available.” Include request_id (copyable in DEBUG).
  • No repeated failing toasts after lock detection.

3) Screen-level spec (what “elegant” means in practice)

Triage deck (cards)

  • Header: contact name + timestamp + direction (inbound/outbound), 1-line summary (scanable).
  • Status row: forcing-state pill + 3-state chip (Ready/Pending/Blocked) + Evidence chip (count + freshness).
  • Reason row: max 2 reason chips + “+N”.
  • Body: transcript excerpt with deterministic highlights (if available); otherwise collapsed “Evidence” sheet entry.
  • Footer (thumb zone): consistent action bar. One primary action + secondary actions; gestures are shortcuts, never the only path.

Picker sheet (anti-anchoring)

  • Two-step commit: tap selects; explicit “Apply” commits (no accidental commits).
  • Fast path: Recents chips + Search; prioritize pick-time p90 over fancy visuals.
  • Suggested: shown as hint-only chip; tap can select but still requires Apply.
  • Truth-forcing copy: disabled Apply explains “Pick a project to teach the system.”

Evidence sheet (“receipts over reasoning”)

  • Show deterministic evidence tokens (count, freshness) and their sources (span_id, interaction_id).
  • When missing, say so explicitly (e.g., “No verified quote anchor; reason: weak_anchor”).
  • Keep it scanable: short excerpts, monospace IDs behind a “Copy” affordance (DEBUG-only if needed).

Thread attribution surface (in-thread learning loop)

  • Each span block shows forcing-state pill + reasons + evidence chip; actions mirror triage semantics.
  • Swipe gestures (if enabled): only as shortcuts; still show visible buttons; haptics on success.
  • Never allow “Confirm” on weak anchors; route through picker-first.

4) Quality bar (iOS platform + mobile doctrine)

  • Touch targets: 44×44pt minimum for every tappable chip/button; icon buttons padded to hit area.
  • Typography: Dynamic Type everywhere (.body, .headline, .footnote), no fixed sizes for primary text.
  • Color: semantic colors; dark mode contrast that survives outdoors; avoid low-opacity gray-on-black for critical info.
  • Gestures: never gesture-only; add visible alternative and short explainer on first encounter.
  • Performance: avoid heavy recomputation per scroll; keep sheets lightweight; prioritize perceived latency < 50ms for taps.

5) Proof gates (VP-required)

Proof What it must show Where it lives
BUILD_PROOF xcodebuild succeeds (sim target), no launch crash. PR comment + CI log pointer (preferred).
SIM_PROOF Chad can open: triage card shows locked confirm + picker-first; real screenshot (not stub). artifacts/ios_simulator_smoke/YYYY-MM-DD/... or PR-embedded images.
REAL_DATA_POINTER Real queue_id/request_id and function version; no secrets. PR description + redacted logs.
GIT_PROOF Merged SHA for the change. PR merge commit.
DB_PROOF (writes) Resolve/dismiss/undo actually mutated the reviewed queue row (audit trail). DATA report with redacted IDs.

This plan intentionally keeps scope narrow: ship picker-first anti-anchoring + receipts + Undo, then iterate with live metrics (pick_time, undo_rate, defect_rate).
