# Camber Doctrine v1 (Guardrails, Red Lines, Heuristics)

Status: DRAFT (derived from vision pack; aligns to canon)
Effective: 2026-02-17
Owner: STRAT

This doc translates the canon into day-to-day decision doctrine.

## Guardrails (non-negotiables)

- Ledger is truth; snapshot is compaction/readback.
- Pointers-or-it-is-not-a-belief: promoted beliefs require receipts.
- Claim-level attribution; multi-project conversations are normal.
- Under uncertainty: HOLD/REVIEW; do not guess.
- No self-reinforcing learning loops.

## Red lines / anti-goals

- Treating a call as a truth container.
- Forcing 1 call = 1 project.
- Promotion without pointers, or pointer hallucination.
- Silent guessing under uncertainty.
- Executing high-cost actions without world contact/authorization.

## Heuristics

Routing
- Prefer avoiding wrong-project promotion, even if it increases review.
- When ambiguous, widen candidates and route to review.

Promotion
- Promotion is a policy boundary (allowlist), not a prompt suggestion.
- Require transcript-span receipts for each promoted item.

Review load
- Optimize to keep Chad’s review queue <= 10 items/day.
- Until stable: bias toward false negatives.

Sharing
- Single ledger with permissioned projections.
- External views show only party-relevant items + receipts.
